# 部署说明（中文）

## 什么时候你才需要部署

先看一句最重要的话：

- 只想自己用，不需要部署
- 想给别人一个网址，再部署

然后再分两种情况：

### 1. 你只是自己本地使用

这种情况不需要部署，直接本地启动就能用：

- 上传报表
- 运行分析
- 查看标签
- 导出结果

### 2. 你想让别人通过网址直接访问

这种情况才需要部署。

也就是说，只有当这个项目要变成“别人打开网址就能直接操作”的系统时，才需要准备服务器、域名和数据库。

## 现在这个项目已经能做什么

当前仓库已经支持本地完整使用：

- 本机启动前后端
- 本机导入示例数据
- 本机上传真实广告报表
- 本机运行 sellerSKU 分析
- 本机导出 CSV 和 Excel

先验证产品可用性，用本地模式就够了。
要对外试用，再进入部署阶段。

## 真要部署，最少需要什么

要做一个“线上可访问版本”，最少需要这些组件：

- 前端
- 后端
- PostgreSQL
- Redis
- HTTPS 反向代理

可以直接理解成这样：

- 前端：给用户打开的网页
- 后端：处理上传、分析、导出
- PostgreSQL：正式数据存储
- Redis：缓存和任务支持
- HTTPS 反向代理：让访问地址稳定且安全

## 推荐的最小生产架构

最小生产架构建议这样配：

- 前端：静态站点
- 后端：FastAPI
- 数据库：PostgreSQL
- 缓存：Redis
- 反向代理：Nginx 或 Caddy

## 部署前一定要改的配置

### 1. 生产环境不要再用 SQLite

SQLite 适合本地演示，不适合正式线上环境。

后端建议改成 PostgreSQL，例如：

```env
DATABASE_URL=postgresql+psycopg2://user:password@host:5432/amazon_ad_workbench
```

### 2. 配置前端 API 地址

前端要知道后端 API 在哪里：

```env
VITE_API_BASE_URL=https://your-api-domain/api
```

### 3. 配置后端允许的前端域名

后端需要正确配置 CORS，否则前端页面会请求失败。

### 4. 开启 HTTPS

线上不要直接裸用 HTTP。

## 最简单的部署方案

### 方案 1：单台 VPS

适合先做一个演示版。

做法通常是：

- 前端打包后放到 Nginx
- 后端用 Uvicorn 跑起来
- PostgreSQL 和 Redis 放本机或托管服务
- 用 Nginx 或 Caddy 做 HTTPS

### 方案 2：前后端分开托管

适合想少做服务器运维。

例如：

- 前端：Vercel / Netlify
- 后端：Railway / Render / VPS
- 数据库：托管 PostgreSQL
- Redis：托管 Redis

## 对新手最实用的建议

不要一上来就追求“完整生产环境”。

更合理的顺序是：

1. 先本地跑通
2. 先用 `Load Demo Data` 看完整流程
3. 再上传两份真实报表验证结果
4. 最后再部署一个线上演示版

## 如果你只想给别人先试用

如果只是想先给别人试用，最实用的做法不是马上上复杂生产环境，而是：

1. 先部署一个演示版
2. 保留 `Load Demo Data`
3. 再让用户自己上传两份报表

这样最容易先把产品体验跑通。

## 部署时要注意什么

- 不要把真实 `API Key` 提交进仓库
- 不要把真实客户报表提交进仓库
- 不要让 Demo 数据在生产环境自动导入
- 不要继续用默认测试配置长期对外
- 最好给后端加日志、监控和备份
