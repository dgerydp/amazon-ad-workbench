# 部署说明（中文）

## 什么时候需要部署

如果只是本地使用，不需要部署。

本项目在本地即可完成：

- 启动前后端
- 上传广告报表
- 运行分析
- 查看标签和导出结果

只有在需要提供一个可公开访问的网址时，才需要进入部署阶段。

## 本地使用与在线使用

### 本地使用

适合：

- 自己验证功能
- 调整规则
- 测试 AI 配置
- 检查报表导入和分析结果

### 在线使用

适合：

- 团队成员通过浏览器共同使用
- 给客户、同事或合作方演示
- 提供固定访问地址

## 在线版本的最小组成

一个可正常访问的线上版本，至少需要以下组件：

- 前端
- 后端
- PostgreSQL
- Redis
- HTTPS 反向代理

对应关系如下：

- 前端：用户访问的页面
- 后端：处理上传、分析、导出和接口请求
- PostgreSQL：业务数据存储
- Redis：缓存与任务支持
- HTTPS 反向代理：对外提供稳定、安全的访问入口

## 推荐的最小部署架构

推荐采用以下结构：

- 前端：静态站点
- 后端：FastAPI
- 数据库：PostgreSQL
- 缓存：Redis
- 反向代理：Nginx 或 Caddy

## 部署前必须调整的配置

### 1. 将 SQLite 切换为 PostgreSQL

SQLite 适合本地演示，不适合正式线上环境。

示例：

```env
DATABASE_URL=postgresql+psycopg2://user:password@host:5432/amazon_ad_workbench
```

### 2. 设置前端 API 地址

前端需要指向正式后端地址：

```env
VITE_API_BASE_URL=https://your-api-domain/api
```

### 3. 配置 CORS

后端需要允许前端域名访问，否则页面请求会被浏览器拦截。

### 4. 启用 HTTPS

线上环境应默认使用 HTTPS，而不是直接暴露 HTTP。

## 常见部署方式

### 方案一：单台 VPS

适合先搭一个演示版或小规模使用版本。

常见做法：

- 前端打包后由 Nginx 提供静态访问
- 后端使用 Uvicorn 运行
- PostgreSQL 和 Redis 使用本机服务或托管服务
- 由 Nginx 或 Caddy 统一处理 HTTPS

### 方案二：前后端分离托管

适合希望减少服务器维护成本的场景。

示例组合：

- 前端：Vercel / Netlify
- 后端：Railway / Render / VPS
- 数据库：托管 PostgreSQL
- Redis：托管 Redis

## 建议的推进顺序

推荐按下面的顺序推进：

1. 先在本地完成基础验证
2. 上传两份真实报表验证结果
3. 再部署一个可访问的版本

## 部署注意事项

- 不要把真实 `API Key` 提交到仓库
- 不要把真实客户报表提交到仓库
- 不要长期使用默认测试配置对外开放
- 建议为后端补充日志、监控和备份
